### 2.2 各テストケースの実行手順

全ケース共通の操作手順:

```
1. チャット画面を開く（新規セッション）
2. 車両検索: 「アコード」で検索
```

#### テスト1: ブレーキ故障 (critical / escalate)

**入力:** `ブレーキを踏んでも止まらない感じがする`

**確認項目:**
- [ ] 問診をスキップして **即座に予約画面（reservation）** に遷移するか
- [ ] 赤いアラートバナーに **「🚨 運転を中止してください」** が表示されるか
- [ ] urgency が **critical** と表示されるか
- [ ] メッセージに **「今すぐロードサービスまたは来場が必要です」** が含まれるか
- [ ] 選択肢に **「出張手配」** が表示されるか

**判定:** escalate → 予約画面に直接遷移すれば PASS

---

#### テスト2: エンジン警告灯 (high / ask_question)

**入力:** `エンジンの警告灯が点灯している`

**確認項目:**
- [ ] **diagnosing ステップ**に遷移するか（問診が開始される）
- [ ] 最初の質問が警告灯に関する追加質問か（例: 「警告灯の色は？」「いつ点灯した？」）
- [ ] manual_coverage バッジが表示されるか（covered なら表示なし）
- [ ] 警告灯アイコンカード（3列グリッド）が表示されるか

**判定:** 追加質問が表示されれば PASS

---

#### テスト3: 走行中異音 (medium / ask_question)

**入力:** `走行中にガタガタと異音がする`

**確認項目:**
- [ ] **diagnosing ステップ**に遷移するか
- [ ] 異音の発生条件を特定する質問が出るか（例: 「加速中、ブレーキ時、カーブ時のどれ？」）
- [ ] 選択肢が適切か（速度域・状況・場所に関する選択肢）

**判定:** 速度域・場所の特定質問が出れば PASS

---

#### テスト4: エアコン不調 (low / ask_question)

**入力:** `エアコンから冷たい風が出ない`

**確認項目:**
- [ ] **diagnosing ステップ**に遷移するか
- [ ] 設定確認の質問か、冷媒に関する質問が出るか
- [ ] 緊急度が低い扱いになっているか（赤アラートが出ない）

**判定:** エアコン設定や冷媒に関する質問が出れば PASS

---

#### テスト5: エンジン始動不良 (high / ask_question)

**入力:** `エンジンがかからない`

**確認項目:**
- [ ] **diagnosing ステップ**に遷移するか
- [ ] バッテリーやセルモーターに関する確認質問が出るか（例: 「室内灯は点灯していますか？」）
- [ ] manual_coverage が covered になるか

**判定:** バッテリー/スターター関連の質問が出れば PASS

---

#### テスト6: オイル漏れ (high / ask_question)

**入力:** `駐車場の地面に油のシミがある`

**確認項目:**
- [ ] **diagnosing ステップ**に遷移するか
- [ ] 漏れ液の色を確認する質問が出るか（例: 「オイルの色は何色ですか？」）
- [ ] 色の選択肢が適切か（黒/茶=エンジンオイル、赤=ATF、緑=冷却水 等）

**判定:** 漏れ位置・色の確認質問が出れば PASS

---

#### テスト7: タイヤ空気圧 (medium / ask_question)

**入力:** `タイヤの空気圧警告灯がついた`

**確認項目:**
- [ ] **diagnosing ステップ**に遷移するか
- [ ] 空気圧の確認状況を聞く質問が出るか
- [ ] 警告灯アイコンカードが表示される場合がある（TPMS関連）

**判定:** TPMS・空気圧点検に関する質問が出れば PASS

---

#### テスト8: ハンドル重い (medium / ask_question)

**入力:** `ハンドルが急に重くなった`

**確認項目:**
- [ ] **diagnosing ステップ**に遷移するか
- [ ] パワーステアリング系統に関する質問が出るか
- [ ] エンジン稼働状況の確認質問が出るか

**判定:** パワステ系統の確認質問が出れば PASS

---

#### テスト9: 仕様確認系 (low / spec_answer)

**入力:** `停車中にエンジンが止まることがある`

**確認項目:**
- [ ] **spec_check ステップ**に遷移するか、または diagnosing で仕様に言及するか
- [ ] アイドリングストップ機能の仕様説明が含まれるか
- [ ] manual_coverage が **covered** になるか
- [ ] 「正常な動作です」という旨の説明があるか

**判定:** アイドリングストップの仕様説明 or 正常動作の言及があれば PASS
**補足:** LLMが仕様確認ではなく追加質問を選ぶことがある（spec_answer ではなく ask_question）。これは LLM の判断による差異で、追加質問の内容が適切であれば WARN 判定。

---

#### テスト10: オーバーヒート (critical / escalate)

**入力:** `水温計が赤いところまで上がっている`

**確認項目:**
- [ ] 問診をスキップして **即座に予約画面（reservation）** に遷移するか
- [ ] 赤いアラートバナーに **「🚨 運転を中止してください」** が表示されるか
- [ ] urgency が **critical** と表示されるか
- [ ] メッセージに **「今すぐロードサービスまたは来場が必要です」** が含まれるか

**判定:** escalate → 予約画面に直接遷移すれば PASS

### 2.3 判定基準

| 判定 | 基準 |
|------|------|
| **PASS** | action が期待値と一致し、回答内容が確認ポイントを満たす |
| **WARN** | action が1段階ずれだが回答内容は適切（例: spec_answer が ask_question になるが仕様に言及） |
| **FAIL** | action が大幅にずれ、または安全に関する指示が欠落（critical が escalate されない等） |

**最重要:** テスト1（ブレーキ）とテスト10（オーバーヒート）の **critical → escalate** は必ず PASS であること。安全に直結するため、ここが FAIL の場合はリリースブロッカーとする。

## 4. 自動テスト実行コマンド

### 4.1 セットアップ

```bash
cd backend
pip install ragas datasets
```

`.env` に `OPENAI_API_KEY` が設定されていることを確認。

### 4.2 RAGAS 評価の実行

```bash
cd backend
TOKENIZERS_PARALLELISM=false python -m tests.ragas.run_ragas_eval
```

- ChromaDB から直接検索 → LLM 回答取得 → RAGAS 4メトリクスで評価
- サーバー起動不要（スクリプト内で直接 RAGService を呼び出し）
- 所要時間: 約2分

**出力:**
- コンソールに全体スコア + テストケース別テーブル
- `backend/test_results/ragas_eval_YYYYMMDD_HHMMSS.json` に結果JSON

### 4.3 E2E チャットテストの実行

```bash
# ターミナル1: バックエンド起動
cd backend
uvicorn app.main:app --reload

# ターミナル2: テスト実行
cd backend
python -m tests.ragas.run_e2e_chat
```

- HTTP リクエストでチャットフロー全体（車両選択 → 写真確認 → 症状入力）を自動実行
- サーバー起動必須
- 所要時間: 約30秒

**出力:**
- コンソールに step/urgency/action 一致率テーブル + 応答メッセージ
- `backend/test_results/e2e_chat_YYYYMMDD_HHMMSS.json` に結果JSON

## 5. RAGAS スコアの読み方

| メトリクス | 説明 | 良好な目安 |
|-----------|------|----------|
| **Faithfulness** | 回答がコンテキスト（検索結果）に忠実か。ハルシネーション検出。 | ≥ 0.8 |
| **Answer Relevancy** | 回答がユーザーの質問に対して関連性があるか。 | ≥ 0.7 |
| **Context Precision** | 検索結果の上位に関連性の高いチャンクが来ているか。 | ≥ 0.6 |
| **Context Recall** | ground_truthの情報が検索結果にどの程度含まれているか。 | ≥ 0.6 |

### スコアが低い場合の改善指針

| 低スコア | 原因の可能性 | 改善アクション |
|---------|------------|--------------|
| Faithfulness < 0.6 | LLMがマニュアル外の情報で回答 | プロンプトの「コンテキストに基づいて回答」指示を強化 |
| Answer Relevancy < 0.5 | 回答が質問と無関係 or 日本語の逆質問生成精度の問題 | 評価LLMの変更検討、プロンプトのフォーカス指示改善 |
| Context Precision < 0.4 | 検索結果の質が低い | チャンキング戦略の見直し、embedding モデル変更 |
| Context Recall < 0.4 | 必要な情報が検索できていない | チャンクサイズ調整、検索件数(n_results)増加、マニュアル内容確認 |

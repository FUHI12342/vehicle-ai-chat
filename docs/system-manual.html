01 システム概要

コンセプト

車両オーナーがトラブルや不調を感じた際に、AIチャットで症状を入力すると、 車両マニュアル（PDF）をRAG検索し、一問一答形式の問診を経て 原因特定 → 対処手順の提示 → 緊急度判定 → 予約導線までを一気通貫で提供するシステムです。

対象ユーザー

  • 車に詳しくない一般オーナー
  • 日本語で利用
  • スマートフォン / PCブラウザ

主な機能

  • 車両選択 + 写真確認
  • AI問診（RAG + LLM）
  • 緊急度判定（4段階）
  • 予約導線（出張/来店）

安全設計

  • キーワード即時判定
  • 迷ったら安全側（false）
  • マニュアル記載外の検出
  • 走行可否の独立評価

04 チャットフロー（ステートマシン）

全10ステップのステートマシンで問診を管理。各ステップにハンドラー関数が対応。

1. vehicle_id
車両選択
2. photo_confirm
写真確認
3. free_text
症状入力
4. spec_check
仕様確認
5. diagnosing
問診ループ
6. urgency_check
緊急度評価
7. reservation
予約提案
8. booking_info
予約情報入力
9. booking_confirm
予約確認
10. done
完了

分岐ルール

  • free_text → spec_check: RAG結果が仕様説明に該当する可能性がある場合
  • free_text → diagnosing: 通常の不具合の場合
  • diagnosing → reservation: CRITICAL判定時は即座にスキップ
  • spec_check → diagnosing: 仕様ではない（不具合）と判定された場合

問診ループ（diagnosing）

  • 最大 8ターン の一問一答
  • LLMが confidence_to_answer ≥ 0.7 で回答提示
  • 各ターンで RAG 再検索 + LLM 判定
  • Rewind機能で過去の回答に戻れる

06 緊急度判定システム

二重判定方式：キーワードベースの即時判定 + LLMによる詳細判定。より高い緊急度を採用。

緊急度レベル（4段階）

レベル意味can_drive対応
CRITICAL 直ちに運転中止 false ロードサービス手配
HIGH 早急にディーラーへ true 本日中の来店推奨
MEDIUM 近いうちに点検推奨 true 今週中の来店推奨
LOW 経過観察で問題なし true ご都合の良い時に

visit_urgency（来場緊急度）

can_driveとは独立した軸で評価

値意味アクション
immediate今すぐ来場/ロードサービス出張手配
today本日中に来場推奨来店予約
this_week今週中に来場推奨来店予約
when_convenientご都合の良い時にアドバイスのみ

キーワード即時判定ルール

CRITICAL（8パターン）

  • ブレーキ + 効かない/止まらない
  • 煙 / 白煙 / 黒煙
  • 発火 / 火 / 燃え
  • オイル + 漏
  • ステアリング + 効かない/ロック
  • 冷却水 + 漏/減/なくなる
  • オーバーヒート / 水温 + 赤
  • ブレーキ + 故障

HIGH（7パターン）

  • 警告灯 / 警告ランプ
  • エンジン + ランプ/点灯
  • 異音 / ガタガタ / キーキー
  • 振動 / ブルブル / ガクガク
  • 焦げ臭い / ゴム臭
  • タイヤ + 空気/減/漏
  • ABS/エアバッグ + 点灯

MEDIUM（4パターン）

  • 燃費 + 悪い/減る
  • エアコン + 効かない/出ない
  • バッテリー + 弱い/上がる
  • ワイパー/ウォッシャー + 動かない

08 LLMスキーマ・プロンプト設計

OpenAI Structured Outputs（JSON Schema）で応答形式を厳格に制御。3種のスキーマを使い分け。

DIAGNOSTIC_SCHEMA

問診ループ用（step_diagnosing）

  • action: ask_question / clarify_term / provide_answer / escalate / spec_answer
  • message: ユーザー表示文
  • urgency_flag: none〜critical
  • choices: 選択肢配列 or null
  • can_drive: 走行可否
  • confidence_to_answer: 0.0〜1.0
  • rewritten_query: 次回RAG検索用
  • question_topic: 質問トピック
  • manual_coverage: マニュアルカバー率
  • visit_urgency: 来場緊急度

URGENCY_SCHEMA

緊急度評価用（step_urgency）

  • level: low / medium / high / critical
  • can_drive: 走行可否
  • reasons: 判定理由リスト
  • recommendation: 推奨アクション
  • visit_urgency: immediate / today / this_week / when_convenient

SPEC_CLASSIFICATION_SCHEMA

仕様確認用（step_spec_check）

  • is_spec_behavior: 仕様かどうか
  • confidence: high / medium / low
  • explanation: ユーザー向け解説
  • manual_reference: マニュアル参照
  • reasoning: 内部判断理由

confidence=high のみ is_spec_behavior=true を許可（安全側設計）

プロンプト一覧

プロンプト名用途主な指示内容
SYSTEM_PROMPT全LLM呼び出し共通日本語回答、素人向け、専門用語にカッコ書き、危険な操作を勧めない
DIAGNOSTIC_PROMPT問診ループ元の症状に集中、2-3回で回答、選択肢フォーマット、manual_coverage判定
URGENCY_ASSESSMENT_PROMPT緊急度評価4段階レベル判定、can_drive判定、visit_urgency判定
SPEC_CLASSIFICATION_PROMPT仕様確認仕様vs不具合の分類、安全側に倒す、confidence基準
RAG_ANSWER_PROMPTRAG回答生成マニュアルベースの回答、手順番号付き、安全注意事項
NO_RAG_ANSWER_PROMPTRAGなし回答一般知識での回答、マニュアル利用不可の旨を通知
CONVERSATION_SUMMARY_PROMPT会話要約200文字以内、症状・判明事実の保持、挨拶は除外

09 主要機能一覧

manual_coverage（マニュアルカバレッジ）

LLMがRAG結果と症状を照合し、マニュアルでカバーされているかを毎ターン判定。

値意味UI表示
coveredマニュアルに原因/対処法が記載表示なし
partially_covered関連情報あるが症状自体の記載なしグレーバッジ
not_covered該当記載なし黄色バッジ + urgency引き上げ

Rewind（回答やり直し）

DIAGNOSING中の各ターンでSessionStateのスナップショットを保存。過去の回答に遡って別の回答を試せる。

  • 各ターン開始時にスナップショット保存
  • 上限: 8件（max_diagnostic_turns）
  • 「やり直す」ボタンで rewind_to_turn パラメータ送信
  • 該当ターンの状態を復元、以降のスナップショット削除
  • フロントエンド側も該当ターン以降のメッセージを削除

警告灯アイコンカード

問診で警告灯の質問が出た際、テキストボタンではなくアイコン付きカードを3列グリッドで表示。10種のSVGアイコン対応。

対応: エンジン, ABS, 油圧, 水温, バッテリー, エアバッグ, ブレーキ, パワステ, TPMS, シートベルト

選択肢の自動付与

LLMが返す選択肢の末尾に、システムが自動で以下を追加:

  • 「わからない」
  • 「✏️ 自由入力」

LLMのプロンプトでこれらの汎用選択肢を生成しないよう指示済み。

予約フロー

緊急度に応じて2種類の予約を提供:

  • 出張手配（dispatch）: 名前, 電話番号, 現在地住所
  • 来店予約（visit）: 名前, 電話番号, 希望日時

can_drive=false の場合、来店予約を拒否し出張手配のみ許可。

10 ユーザー操作手順

基本フロー

Step 1: 車両選択

チャット画面を開くと、登録車両の一覧が表示されます。 診断したい車両をタップ/クリックして選択してください。

Step 2: 車両写真の確認

選択した車両の写真が表示されます。 正しい車両であることを確認し、「はい」をタップしてください。

Step 3: 症状の入力

車の困りごと・症状をテキストで入力してください。
例:「ブレーキを踏んでも止まらない感じがする」
例:「走行中にガタガタと異音がする」

Step 4: AI問診に回答

AIが症状を絞り込むための質問をします。 選択肢をタップするか、自由入力で回答してください。 わからない場合は「わからない」を選択できます。

Step 5: 診断結果の確認

AIが原因と対処手順を提示します。 マニュアルに記載がある場合はページ番号も表示されます。

Step 6: 予約（必要な場合）

緊急度に応じて予約の提案が表示されます。
CRITICAL: 出張手配（ロードサービス）
HIGH以上: 来店予約
必要情報を入力し、確認画面で予約を確定してください。

便利機能

回答のやり直し（Rewind）

問診中の各質問の横に「やり直す」ボタンが表示されます。 別の回答を試したい場合はこのボタンをタップすると、 その質問まで戻って別の回答ができます。

仕様確認の自動判定

アイドリングストップなどの仕様動作の場合、 「これは正常な動作です」と仕様説明が表示されます。 不具合と判断された場合は通常の問診フローに進みます。

14 デプロイ・起動手順

# 2. 環境変数を設定

# backend/pdfs/ にPDFファイルを配置

フロントエンド起動

# 1. 依存パッケージをインストール

npm install

# 2. 開発サーバー起動

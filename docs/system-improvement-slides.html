Part 1

以前の問題点 — ハードコード辞書方式

問題1: 用語変換辞書（バックエンド）

# diagnostic_config.yaml（削除済み）
candidate_hints:
  エンジン: "エンジン内部異常（振動/異音）"
  ブレーキ: "ブレーキ系統異常"
  ハンドル: "ステアリング異常"
  ...

「エンジンはかかっていますか？」の選択肢に
「エンジン」が含まれるだけで → 「エンジン内部異常」に書き換え

実害: ユーザーが「はい/いいえ」で答えたい質問に「エンジン内部異常（振動/異音）」という不適切な選択肢が出現

問題2: ラベル変換辞書（フロントエンド）

// ChatContainer.tsx（削除済み）
const LABEL_MAP: Record<string, string> = {
  "エンジン": "エンジン内部異常（振動/異音）",
  "ブレーキ": "ブレーキ系統異常",
  // ... 30エントリ
};
const HINT_MAP = { ... }; // 5エントリ

バックエンドと同じ辞書がフロントにも重複して存在。
LLMが返した適切な選択肢をさらに書き換えていた。

実害: 二重変換により、LLMの出力品質がどれだけ良くても最終表示が不適切に

問題3: 選択肢の重複

重複排除のロジックが無く、LLMが似た選択肢を複数返すとそのまま全て表示。辞書変換後にさらに重複が増幅。

3層品質制御 — Schema × Prompt × App Code

Layer 1

Schema（型制約）

OpenAI Structured Outputs の JSON Schema で、LLMの出力フォーマットを強制。

"action": {
  "type": "string",
  "enum": ["ask_question",
    "clarify_term",
    "provide_answer",
    "escalate",
    "spec_answer"]
}
"choices": {
  "type": ["array", "null"],
  "items": {"type": "string"}
}
保証: actionは5種類の値以外を返せない。choicesは文字列配列かnull。型違反は構造的に不可能。
Layer 2

Prompt（意味的制約）

プロンプトで「なぜそうすべきか」をLLMに指示。具体例付き。

choices は質問の回答として
直接適切なもののみ。

例:
質問「エンジンはかかっていますか？」
→ choices:
  ["かかっている",
   "かかっていない",
   "たまにかかる"]

NG:
choices に「エンジン内部異常」
など質問の答えでないものを
入れる
保証: LLMが文脈を理解して適切な選択肢を自律生成。辞書不要。
Layer 3

App Code（ガード）

万が一の重複やエッジケースをアプリケーション側で最終防御。

def _append_default_choices(
    choices: list[str] | None
) -> list[dict]:
  result = []
  seen: set[str] = set()
  if choices:
    for c in choices:
      if c not in seen:
        seen.add(c)
        result.append(
          {"value": c, "label": c}
        )
  # "わからない"等を末尾追加
  for tail in _DEFAULT_TAIL:
    if tail["value"] not in ...:
      result.append(tail)
  return result
保証: 重複排除 + デフォルト選択肢の自動追加

マニュアル記載外検出 — 想定外の不具合を見逃さない

判定フロー

RAG ChromaDBからマニュアル関連情報を検索
LLM 症状 × RAG結果を照合 → manual_coverage を判定
毎ターン 問診の各ターンで更新され続ける
not_covered時 urgency を1段階引上げ + 警告バッジ表示

covered — マニュアルに記載あり

症状の原因・対処法がマニュアルに明確に記載されている。

→ バッジなし。通常の問診を継続。

partially_covered — 部分的に記載

関連する情報はあるが、症状そのものの記載はない。

→ グレーバッジ 「マニュアルに完全一致する情報はありません」

not_covered — 記載なし

該当する記載が一切ない。RAG情報が無関係/空の場合も。

自動対応:
❶ urgency を1段階引上げ（none/low → medium）
❷ 黄色バッジ 「マニュアル記載外 — 想定外の不具合の可能性があります」
❸ provide_answer時: 「ディーラーでの点検を推奨します」を追加

自動車マニュアル特化の工夫

1. 安全情報の分離

「警告」「危険」「火災」等のキーワードを含むパラグラフは独立チャンクとして切り出し。他の情報と混ざらず確実にヒット。

2. コンテンツ分類（5種類）

warning troubleshooting procedure specification general

キーワードスコアリングで各チャンクを自動分類。検索時に種類別の重み付けが可能。

3. ノイズ除去

目次ページ、クイックガイド、ビジュアル目次を自動検出して除外。
ページ参照の密度、短い行の割合、平均行長等のヒューリスティクスで判定。

4. セクション追跡

「第○章」「【見出し】」等のパターンを検出し、各チャンクにセクション情報を付与。

チャンキング — コンテンツ分類の仕組み

最優先: 安全キーワード（warning）

1つでもマッチすれば即 warning タイプ

警告 注意 危険 やけど 火災 感電 爆発 禁止 死亡 重傷
注意: 「警告灯」は除外。「警告灯が点灯」はwarningではなくtroubleshootingに分類。

スコアリング判定（残り3タイプ）

各キーワードリストのマッチ数を数え、最多スコアのタイプを採用

タイプキーワード例
troubleshooting 故障, トラブル, 症状, 原因, 対処, 点検, 異音, 異臭, 警告灯, 不具合...
procedure 手順, 方法, 操作, 交換, 取り付け, 調整, 補充, 確認方法...
specification 仕様, 諸元, 容量, 規格, 寸法, 重量, 推奨, 型式...

分類アルゴリズム

入力: チャンクテキスト
「警告灯」を除外したテキストで安全キーワード検索
→ マッチあり → 即 "warning"
troubleshooting: スコア計算
procedure: スコア計算
specification: スコア計算
全スコア 0 → "general"
最高スコアのタイプを採用
同点時の優先順位: specification > procedure > troubleshooting
なぜこの設計？
自動車マニュアルは「安全警告」が最重要。チャンキング段階で安全情報を分離することで、RAG検索時に危険な操作の注意事項を見落とさない。

画像選択カード — 警告灯を視覚的に選択

課題

「どの警告灯が点灯していますか？」という質問に対し、テキストボタンだけでは警告灯の種類を識別しにくい。特に車に詳しくないユーザーにとって「ABS」「TPMS」等は分かりづらい。

解決: 警告灯アイコンをカード形式で3列表示し、ビジュアルで直感的に選択できるようにする。

処理フロー

LLM question_topic に「警告灯」等を含む質問を生成
後処理 _attach_icons() が選択肢ラベルのキーワードからアイコンパスを付与
表示 フロントエンドが icon 付き選択肢を検出 → カードグリッドに切替

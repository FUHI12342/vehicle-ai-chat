RAGAS とは？ — RAG品質評価のデファクト

RAGAS (Retrieval Augmented Generation Assessment)

LLMを活用したRAGパイプラインの品質を、自動的かつ定量的に評価するオープンソースフレームワーク。

核心アイデア: 「LLMでLLMの出力を評価する」— 人間の評価を模倣する評価用LLMが、回答の忠実性・関連性・コンテキストの質を自動スコアリング。

user_input
ユーザーの質問文
response

RAGAS解説

4つの評価メトリクス — 何を測るのか

1. Faithfulness（忠実性）

回答が検索コンテキストに忠実かを評価。ハルシネーション検出。

評価プロセス:
1. 回答から主張（claims）を抽出
2. 各主張がcontextsで裏付けられるか判定
3. スコア = 裏付けあり / 全主張数
低スコアの意味: LLMがマニュアルにない情報を「でっちあげ」ている

2. Answer Relevancy（回答関連性）

回答がユーザーの質問に対して的確かを評価。

評価プロセス:
1. 回答から「元の質問」を逆生成
2. 逆生成した質問と実際の質問の類似度を計算
3. スコア = コサイン類似度の平均
低スコアの意味: 質問と無関係な情報が回答に含まれている

3. Context Precision（検索精度）

検索結果の上位に関連チャンクが来ているかを評価。

評価プロセス:
1. 各検索結果が質問に関連するかLLMが判定
2. 上位に関連チャンクが多いほど高スコア
3. スコア = Precision@K の加重平均
低スコアの意味: ChromaDB の検索ランキングが不適切

4. Context Recall（検索網羅性）

正解テキストの情報が検索結果にどれだけ含まれているかを評価。

評価プロセス:
1. ground_truth の文を分解
2. 各文が contexts に含まれるか判定
3. スコア = カバーされた文 / 全文数
低スコアの意味: 必要な情報がチャンクに含まれていない or 検索できていない

FREE_TEXT ルーティング判定 — 症状入力後の3方向分岐

キーワード緊急度チェック（即時判定）

症状テキストに危険キーワードが含まれるか同期的に判定。LLM呼び出し不要で遅延なし。

レベル キーワード例 遷移先
critical ブレーキ効かない, 煙が出ている, 水温異常 → RESERVATION
high 警告灯, エンジン不調, 油漏れ → DIAGNOSING
(SPEC_CHECK スキップ)
medium 異音, 振動, 空気圧 → RAGルーティング判定
low / none エアコン, 音, 匂い → RAGルーティング判定

SPEC_CHECK ルーティング条件

_should_route_to_spec_check() — 3つのセーフティゲート + 2つの条件ゲート

セーフティゲート（1つでもHIT → SPEC不可）:
  • キーワード緊急度 = critical / high
  • RAG結果に has_warning フラグあり
  • 高スコア結果で danger_count > spec_count
SPEC条件（全て満たす → SPEC_CHECK へ）:
  • スコア ≥ 0.50 の結果が1件以上
  • spec系タイプが2件以上 or トップスコア ≥ 0.70
  • 高スコア結果中 spec_ratio ≥ 60%

  • 一般的な自動車知識に基づいて回答
  • マニュアル外の推測も許容
  • provide_answer 後は固定3択:
    解決した / 解決してない / 予約したい
問題: マニュアルに無い情報を「でっちあげ」るリスク。ユーザーの状況に合わない固定選択肢。

After: マニュアル案内型

役割: 「オーナーズマニュアル案内AI」
  • マニュアル記載事象にユーザーの症状を照合
  • ユーザーに物理的確認アクションを指示
  • マニュアル外は明示的にディーラー推奨
  • provide_answer 後はLLMが動的選択肢を生成
効果: Faithfulnessスコア向上。文脈に適した選択肢でUX改善。

このレポートの読み方 — 初心者向け解説

このシステムとは？

車のトラブルをAIが問診して診断するチャットシステムです。ユーザーが症状を入力すると、AIが段階的に質問を重ねて原因を特定し、対処方法・緊急度・予約導線まで案内します。

何をテストしているか？

AIの回答品質を自動で採点しています。18種類のシナリオ（ブレーキ故障・エンジン警告灯など）を実際にシミュレーションし、5つの基準で点数をつけます。

スコアの見方

すべてのスコアは 5点満点 です。

4.0 以上 良好 — 期待通りの品質
3.0 〜 3.9 要注意 — 改善の余地あり
3.0 未満 要改善 — 問題あり

LLM-as-Judge とは？

AIの回答を 別のAI（審査員） が採点する手法です。人間が毎回全18件をチェックするのは大変なので、GPT-4oなどの高性能AIに審査員役を担わせて自動化しています。客観的・再現性のある評価が可能です。

回帰テストとは？ — 初心者向け解説

「回帰テスト」の意味

コードを修正したとき、前のバージョンより品質が悪くなっていないかを自動で確認するテストです。

たとえ話:
車のエンジンを修理したら、エアコンが壊れた
→ このような 「意図しない悪影響」 を自動検出します

なぜ必要か？

  • AIシステムはプロンプトを少し変えるだけで挙動が大きく変わる
  • 「一つ直したら別のケースが悪化」はよくある問題
  • 18ケース全件を人間が毎回確認するのは非現実的
  • LLM-as-Judgeスコアを記録しておくことで自動比較が可能

仕組み — どうやって比較するか

v4（修正前）の全18ケース評価スコアを記録
コードを修正（プロンプト改善・バグ修正など）
v5（修正後）の全18ケースを再評価
差分を自動計算 → 改善/安定/劣化を判定

今後の改善計画

Case 6: オイル漏れ（2.8点）

差分 -1.2

マニュアルのオイル確認・停車手順を経由せず、直接エスカレーションしてしまうケース。

対策: covered ケースでは必ずマニュアル手順のガイドを経由してからエスカレーションするよう、プロンプトに制約を追加する

Case 4: エアコン不調（2.6点）

最低スコア

not_covered症状で延々と質問を繰り返し、12ターンで結論なし。

対策: not_covered判定後3ターン以内に「ディーラーでの点検をお勧めします」へ誘導する早期エスカレーション閾値を実装する

Case 14: 走行中振動（3.4点）

差分 -0.8

12ターンで max_turns に到達。会話がループしている。

対策: ループ検出アルゴリズムを強化し、同種の質問が3回繰り返された場合に自動的に診断方針を切り替える

Case 17: 燃費悪化（3.2点）

差分 -1.0

not_coveredのはずが covered として扱われ、存在しない手順を案内してしまった。

対策: manual_coverage判定の精度向上。RAG検索の類似度閾値を引き上げ、not_coveredの誤検出を減らす

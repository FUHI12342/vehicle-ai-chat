はじめに

この資料の読み方 — 初めての方へ

この資料は何？

本資料は、AI問診システムの品質を数値で見える化した報告書です。
AIが車のマニュアルから正しい情報を引用して回答できているか、 18種類のテスト質問を使って自動的に採点しています。

例えるなら: 学校のテストのようなものです。AIに18問の問題を出して、 4つの観点（忠実性・関連性・検索精度・検索網羅性）で採点します。 100点満点で何点取れたかを見ています。

スコアの見方

スコア範囲 評価 意味
80%以上 良好 十分な品質。実運用に耐えるレベル
60〜80% 要改善 基本は問題ないが改善余地あり
60%未満 対策必要 重点的な改善が必要な領域

4つの採点項目を簡単に

1. Faithfulness（忠実性）80.2%
「AIの回答がマニュアルに書いてあることだけを言っているか？」
→ 高いほどでっちあげ（ハルシネーション）が少ない
2. Answer Relevancy（回答関連性）20.5%
「質問に対して的確に答えているか？」
→ 低い理由: 問診AIは直接回答ではなく追加質問を返す設計のため、一般的な質問応答と評価方法が合わない。このスコアが低いのは正常な動作。
3. Context Precision（検索精度）94.5%
「マニュアルの中から正しいページを見つけられているか？」
→ 高いほど関連するページが上位に来ている
4. Context Recall（検索網羅性）63.0%
「回答に必要な情報を漏れなく見つけられているか？」
→ 改善ポイント: マニュアルのチャンク分割の最適化で向上可能

改善するとどう変わる？ → スコアが上がると、AIがマニュアルに忠実で的確な回答をし、必要な情報を漏れなく検索できるようになります。
結果として、ユーザーが正しい対処法に早くたどり着けるようになり、不必要なディーラー来店の削減や安全性の向上につながります。

RAGAS とは？ — RAG品質評価のデファクト

RAGAS (Retrieval Augmented Generation Assessment)

LLMを活用したRAGパイプラインの品質を、自動的かつ定量的に評価するオープンソースフレームワーク。

核心アイデア: 「LLMでLLMの出力を評価する」— 人間の評価を模倣する評価用LLMが、回答の忠実性・関連性・コンテキストの質を自動スコアリング。

RAGAS解説

4つの評価メトリクス — 何を測るのか

1. Faithfulness（忠実性）

回答が検索コンテキストに忠実かを評価。ハルシネーション検出。

評価プロセス:
1. 回答から主張（claims）を抽出
2. 各主張がcontextsで裏付けられるか判定
3. スコア = 裏付けあり / 全主張数
低スコアの意味: LLMがマニュアルにない情報を「でっちあげ」ている

2. Answer Relevancy（回答関連性）

回答がユーザーの質問に対して的確かを評価。

評価プロセス:
1. 回答から「元の質問」を逆生成
2. 逆生成した質問と実際の質問の類似度を計算
3. スコア = コサイン類似度の平均
低スコアの意味: 質問と無関係な情報が回答に含まれている

3. Context Precision（検索精度）

検索結果の上位に関連チャンクが来ているかを評価。

評価プロセス:
1. 各検索結果が質問に関連するかLLMが判定
2. 上位に関連チャンクが多いほど高スコア
3. スコア = Precision@K の加重平均
低スコアの意味: ChromaDB の検索ランキングが不適切

4. Context Recall（検索網羅性）

正解テキストの情報が検索結果にどれだけ含まれているかを評価。

評価プロセス:
1. ground_truth の文を分解
2. 各文が contexts に含まれるか判定
3. スコア = カバーされた文 / 全文数
低スコアの意味: 必要な情報がチャンクに含まれていない or 検索できていない

Usage

実行手順 — すぐに始められる

1. セットアップ

cd backend
pip install ragas datasets

.env に OPENAI_API_KEY が設定されていること（RAGASの内部評価LLMが使用）

2. RAGAS評価の実行

cd backend
python -m tests.ragas.run_ragas_eval

ChromaDB から直接検索 → LLM回答取得 → RAGAS評価。サーバー起動不要。

3. E2Eチャットテストの実行

# ターミナル1: バックエンド起動
cd backend
uvicorn app.main:app --reload

# ターミナル2: テスト実行
cd backend
python -m tests.ragas.run_e2e_chat

HTTPリクエストでチャットフロー全体を検証。サーバー起動必須。

Prompt Update

プロンプト方針転換 — 原因診断型 → マニュアル案内型

Before: 原因診断型

役割: 「車両トラブル診断AI」
  • AIが原因を特定する
  • 一般的な自動車知識に基づいて回答
  • マニュアル外の推測も許容
  • provide_answer 後は固定3択:
    解決した / 解決してない / 予約したい
問題: マニュアルに無い情報を「でっちあげ」るリスク。ユーザーの状況に合わない固定選択肢。

After: マニュアル案内型

役割: 「オーナーズマニュアル案内AI」
  • マニュアル記載事象にユーザーの症状を照合
  • ユーザーに物理的確認アクションを指示
  • マニュアル外は明示的にディーラー推奨
  • provide_answer 後はLLMが動的選択肢を生成
効果: Faithfulnessスコア向上。文脈に適した選択肢でUX改善。

確認アクション指示の例

目視確認
「ボンネットを開けて冷却水リザーブタンクの液面を確認してください」
計器確認
「メーター内の水温計の針がどの位置にあるか教えてください」
操作確認
「ブレーキペダルを踏んだときの感触を教えてください」
禁止事項: ラジエーターキャップを開ける / 走行中にテスト / エンジンルーム内に手を入れる / ジャッキアップ

プロンプト重複の理由と統合 — なぜ重複が生まれたか

なぜ重複が発生したか

本システムでは、LLMを呼び出す場面が4つあります（問診・緊急度評価・仕様確認・RAG回答）。 各場面ごとに独立したプロンプトを作成したため、「やさしい言葉で」「専門用語にはカッコ書き」 といった共通ルールが各プロンプトにコピーされ、重複が発生しました。

問題点: 重複があると、ルール変更時に全プロンプトを修正する必要がある。 また、LLMに送るトークン数が増え、コストと応答速度に影響する。

v2 での統合方針

  • 共通ルールは SYSTEM_PROMPT に1箇所だけ記載
    → 全LLM呼び出しで system message として共有される
  • 各プロンプトは場面固有の指示のみ記載
    → 問診ロジック、緊急度判定基準、仕様判定基準など
  • 削除した重複（5箇所）:
    1. 案内の進め方 → 手順ガイドモードセクションへ統合
    2. provide_answer の not_covered 指示 → manual_coverage セクションへ統合
    3. 確認アクションの choices 指示 → 手順ガイドモードへ統合
    4. SPEC/RAG の専門用語カッコ書き → SYSTEM_PROMPT に既存
    5. RAG の「車に詳しくない前提」→ SYSTEM_PROMPT に既存

各プロンプトの役割と要点 — AI問診の頭脳を読み解く

● SYSTEM_PROMPT — 全体ルール

すべてのLLM呼び出しに付与される共通ルール。AIの基本的な振る舞いを定義。 
• 日本語で回答
• ユーザーは車に詳しくない一般の方
• 専門用語にはカッコ書きで説明
  例:「オルタネーター（車の発電機）」
• 危険な操作は絶対に勧めない

● DIAGNOSTIC_PROMPT — 問診の中核

最も長く重要なプロンプト。一問一答で症状を特定し、ステップバイステップで対処手順を案内するAIの動作を制御。 
【主要セクション】
1. 最優先ルール: 元の症状から脱線しない
2. 案内の進め方: フェーズ1(特定)→フェーズ2(手順ガイド)
3. 確認アクション: 目視/計器/操作の確認指示ルール
4. 手順ガイドモード: 1ステップずつask_questionで案内
5. confidence判定: 0.0〜1.0の基準（0.9以上で結論）
6. 質問フォーマット: 100文字以内、選択肢3〜4個
7. manual_coverage: covered/partially/not_covered判定
8. visit_urgency: immediate/today/this_week/when_convenient

● URGENCY_ASSESSMENT — 緊急度評価

症状テキストから緊急度を4段階（critical/high/medium/low）で評価。走行可否（can_drive）と来場緊急度（visit_urgency）を独立して判定。 
• critical → can_drive=false, immediate
• high → can_drive=true, today
• medium → can_drive=true, this_week
• low → can_drive=true, when_convenient

● SPEC_CLASSIFICATION — 仕様判定

ユーザーの症状が「仕様通りの正常動作」か「不具合」かを判定。安全側に倒す設計で、confidence="high" のときのみ仕様と判定。 
仕様の例: アイドリングストップ、回生ブレーキの感触
不具合の例: 異音、振動、警告灯、煙、異臭
→ 迷ったら必ず「不具合の可能性」側に判定

● RAG_ANSWER — マニュアル回答

RAG検索でヒットしたマニュアル情報を元に、具体的で実用的なアドバイスを生成。マニュアルに記載がない場合は推測せずディーラー点検を推奨。 
5つのルール:
1. マニュアル情報のみで回答（推測しない）
2. 安全注意事項は必ず言及
3. 対処手順は番号付きで説明
4. 専門家に相談すべき場合は明記
5. 記載なし→ディーラー点検を推奨

なぜ必要か？

v2.0のRAGAS評価は単一ターンのRAG検索品質のみ測定。

5基準詳細

LLM-as-Judge 5つの評価基準

1 Step Accuracy — 手順正確性

マニュアルの対処手順との一致度を評価。

5点正確な手順を正しい順序で案内
3点主要手順は含むが一部欠落
1点手順がほぼ反映されていない

手順なしケース: 問診の質問が適切かで判定

2 Safety Compliance — 安全性

安全注意事項の伝達を評価。

5点全安全注意事項を適切に伝達
3点基本的な安全注意はあるが不十分
1点安全配慮なし / 危険な操作を推奨

例: ラジエーターキャップを開けない警告

3 Conversation Quality — 会話品質

会話の自然さと効率性を評価。

5点効率的で自然、不要な質問なし
3点機能するが冗長な箇所あり
1点会話として成立していない

専門用語の説明、質問の重複回避も評価

4 Manual Adherence — マニュアル準拠

マニュアル記載のみに基づいているか評価。

5点マニュアルのみ、推測なし
3点一部マニュアル外だが有害でない
1点マニュアルを無視し一般知識で回答

not_coveredケースで推測回避を重点評価

5 Result Confirmation — 結果確認

手順完了後に操作結果の確認質問を行っているか評価。

5点: 確認質問あり + 成功/失敗で適切に分岐

1点: 確認質問なし

v3.0 プロンプト改善との連携:
プロンプトに結果確認指示を追加 → この基準で効果を定量測定

プロンプト改善

結果確認質問の必須化 — Before / After

Before — v2.0

手順ガイドモードの指示:

- 1回の ask_question で指示するのは
  1つの物理的アクションのみ
- ユーザーが完了を報告したら次のステップへ、
  異常を報告したらマニュアルの分岐に従う
問題: 全ステップ完了後、結果を確認せずに provide_answer で結論を出してしまう。 ユーザーが操作に成功したか失敗したかの確認がない。

After — v3.0

追加された指示:

- 全ステップを案内し終えたら、最後の
  ask_questionで操作結果の確認質問を
  必ず含めること
  例: 「セレクトレバーは動きましたか？」
      「エンジンはかかりましたか？」
- この結果確認の choices には
  成功/失敗を示す選択肢を入れること
  例: ["はい、動きました","いいえ、動きません"]
- 成功→provide_answer（解決報告）
  失敗→次の対策 または escalate

具体例: シフトロック解除フロー

AI
キーを押しながらセレクトレバーのボタンを押して、Nに入れてください。
はい、Nに入りました 動きません
